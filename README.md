## 📐 核心設計心法：狀態管理與職責分離

在開發中，我嚴格遵循 **「UI 畫面」與「業務邏輯（Business Logic）」徹底切開** 的原則（Clean Architecture 思維）。

### 1. 高頻率與即時數據流處理 (如：車輛 GPS 追蹤、狀態即時更新)
麵對需要即時監聽數據的情境（例如動態演算法 API 對接、車輛位置變更），我習慣運用狀態管理的異步通知機制（如 `AsyncNotifier` 或 `StreamProvider`）來建立強健的數據流，確保：
- **極致流暢的 UI 渲染**：避免不必要的 Widget 全局 Rebuild，將記憶體與耗電降到最低，防止細軟直髮……不對，是防止系統架構「塌掉」與卡頓。
- **異步狀態的安全掌控**：優雅處理 Data、Loading、Error 三種狀態，提升用戶體驗。

### 2. 跨平台審查（Review）應變經驗
具備雙平台商店上架經驗，熟悉 Apple 與 Google 的審查規範。面對突發的退件（Rejection），能快速定位是隱私權政策、硬體權限（如 GPS 後台定位權限）還是 UI 佈局衝突，並在最短時間內修正重新送審。

## 💬 團隊協作與危機處理 (Hotfix)

### 1. 良好的溝通往往比 Git 衝突解決更加重要
技術上的 Code 衝突可以用工具解，但邏輯與規格的衝突必須靠透明、即時的通訊。我習慣在專案初期即與 PM、後端工程師主動對齊 API 規格與狀態機定義，大幅降低後期的通訊成本。

### 2. 生產環境 (Production) 高壓排查
當線上環境發生緊急 Live Bug（例如特定機型閃退或金鑰失效）時，我有一套穩健的處理 SOP：
> **「先觀察現象（撈取 Log/Crashlytics） ➡️ 定位問題結構 ➡️ 將複雜問題拆解 ➡️ 精準進行 Hotfix 容錯處理」**
確保在極短時間內穩住用戶體驗底線，不盲目通靈。
